# Post-mortem: падение прода 2026-06-19 (пример «как делать нельзя»)

Этот документ — зафиксированный реальный кейс. Цель: чтобы команда (и нейросети) на него ссылались и **не повторяли** ошибок. Читать вместе с `RULES.md`.

## Кратко (TL;DR)
Прод (`bot.knzteam.ru`) ушёл в **502** — `app` крутился в рестарт-петле, потому что подключался к MySQL как **пустой пользователь** (`Access denied for user ''`). Причина оказалась простой (старый env в контейнере), но её **чуть не «починили» командой, стирающей базу**. Починили правильно — пересозданием контейнеров, **без потери данных**.

## Контекст
Проект передан коллеге для самостоятельной разработки по схеме `dev → PR → master → автодеплой`. Коллега (через Codex) поверх нашей системы внёс ряд изменений в **инфраструктуру** (а это не его зона):
- переписал `cd.yml`: вместо простого деплоя стал **генерировать `.env` на сервере из GitHub Secrets**, с фолбэком «если секретов нет — оставить старый `.env`»;
- добавил `migrate.js` и запуск `node scripts/migrate.js && node src/server.js` (миграции **блокируют** старт app);
- добавил `schema-checks.js`;
- в `compose.prod.yml` ввёл **вложенные дефолты** вида `${MYSQL_USER:-${DB_USERNAME:-}}`.

## Симптомы
- `curl https://bot.knzteam.ru/health` → **HTTP 502** (nginx жив, upstream :3000 мёртв).
- `docker compose ps` → `app` и `worker` в **`Restarting (1)`**, `db` — `healthy`.
- Логи app: `Migration failed: Access denied for user ''@'172.18.0.x' (using password: YES)` — коннект как **пустой юзер**.

## Расследование и ВАЖНАЯ развилка
Первичный (ошибочный) диагноз в другом чате: «`MYSQL_USER` пустой, прикладного пользователя в БД нет» → предложение `docker compose down -v` (пересоздать том). **Это стёрло бы все данные регистраций.**

Проверка по фактам показала обратное:
- в `.env` `MYSQL_USER/PASSWORD/DATABASE/ROOT` — **заданы**;
- пользователь `megapolis_bot_user@%` в MySQL — **существует**;
- база `megapolis_bot` — **существует**;
- прямой тест `mysql -u<app_user> -p<pass>` → **успех** (`app_login_ok = 1`);
- `docker compose -f compose.prod.yml config` → `DB_USERNAME` резолвится в `megapolis_bot_user` **корректно**.

**Истинная причина:** контейнер `app` был создан во время более раннего битого деплоя (когда `.env` был неполный) и **не пересоздавался** после того, как `.env` починили. То есть в работающем контейнере остался **старый/пустой `DB_USERNAME`**, хотя файл `.env` на диске уже верный.

## Как починили (без потери данных)
```bash
cd /opt/megapolis-tg-bot
docker compose -f compose.prod.yml up -d --force-recreate
```
Контейнеры пересоздались с актуальным `.env` → `migrate.js` подключился как `megapolis_bot_user` → миграции прошли → app поднялся. `health` → `ok: true`, данные на месте.

## Корневые причины
1. **Полу-готовая secrets-модель.** `cd.yml` генерил `.env` из GitHub Secrets, которых не было → срабатывал фолбэк «оставить старый `.env`». Это **маскировало** проблему и приводило к запутанным сбоям вместо честного падения.
2. **Миграции блокируют старт.** `migrate.js && server.js`: любая проблема с БД роняет **весь app** в рестарт-петлю и кладёт прод.
3. **Несколько владельцев инфры.** Одновременно её меняли коллега (Codex) и два чата с ИИ — «наслоения» и рассинхрон.
4. **Диагностика по догадке.** Предложение `down -v` без проверки реальной ошибки едва не стёрло данные.

## Уроки (что зафиксировано в правилах)
- **Никогда** не «чинить» подключение к БД через `docker compose down -v` — это стирает том. Сначала **прочитать реальную ошибку** из логов и проверить факты (есть ли юзер, логинится ли он).
- **Инфру (CI/CD, compose, Dockerfile) меняет только владелец.** Теперь это закреплено на уровне git (`CODEOWNERS` + require code owner review) — см. `RULES.md`.
- **Секреты — в ручном `.env` на сервере**, не в GitHub. Для проекта такого размера генерация `.env` из секретов избыточна и хрупка. `cd.yml` упрощён до простого деплоя.
- **Миграции не должны блокировать старт app** так, чтобы он уходил в рестарт-петлю (рекомендация на доработку — идемпотентный отдельный шаг/ретраи, app поднимается даже при временной недоступности БД).
- **Один владелец инфры за раз.** Не запускать параллельные правки одних файлов из разных мест.
- При деплое всегда **пересоздавать** контейнеры так, чтобы они брали актуальный `.env`.

# Bitrix24 CRM integration plan

Цель: передавать в Bitrix24 людей из Telegram-бота и их регистрации на мероприятия, чтобы в CRM было видно:

- человек просто заполнил анкету в боте;
- человек выбрал мероприятие;
- формат участия: онлайн или офлайн;
- статус офлайн-заявки: на модерации, подтвержден, отклонен;
- факт прихода на офлайн-мероприятие.

## Рекомендованная модель Bitrix24

Так как в Bitrix24 уже создана отдельная воронка сделок, основной объект интеграции должен быть `Сделка`, а не устаревший объект `Лид`.

В Bitrix24 REST для сделки используется:

- современный метод `crm.item.add` с `entityTypeId = 2`;
- либо старый `crm.deal.add`, но документация Bitrix24 помечает его как deprecated.

Рекомендация: использовать `crm.item.add`.

Контакт лучше создавать/обновлять отдельно:

- один `Контакт` на человека;
- одна `Сделка` на регистрацию человека на конкретное мероприятие;
- для состояния “просто зарегистрировался в боте” можно создать отдельную сделку в первой стадии без события или хранить это только в контакте. Практичнее для воронки - создать сделку в стадии “Анкета заполнена”.

## Что нужно получить из Bitrix24

### Авторизация

Для первой версии достаточно входящего webhook Bitrix24 с правами CRM.

Нужно прислать:

```text
BITRIX24_WEBHOOK_URL=https://your-domain.bitrix24.ru/rest/{user_id}/{webhook_key}/
```

Webhook должен быть создан от пользователя, у которого есть права:

- читать CRM;
- создавать/изменять контакты;
- создавать/изменять сделки;
- читать воронки и стадии.

### Воронка сделок

Нужно узнать `categoryId` воронки. Метод:

```text
crm.category.list
entityTypeId = 2
```

### Стадии

Нужно узнать `stageId` стадий этой воронки. Метод:

```text
crm.status.list
filter[ENTITY_ID] = DEAL_STAGE_{categoryId}
```

Желаемая структура стадий:

- `Анкета заполнена` - человек оставил контакты, но еще не выбрал событие.
- `Выбрано мероприятие` - событие выбрано, формат еще не финализирован.
- `Онлайн зарегистрирован` - онлайн без модерации.
- `Офлайн на проверке` - офлайн-заявка ждет модератора.
- `Офлайн подтвержден` - модератор подтвердил.
- `Пришел` - отметка ресепшна.
- `Отклонен / не пришел / отменен` - финальные негативные статусы.

Точные коды стадий будут вида:

```text
C{categoryId}:NEW
C{categoryId}:...
```

Их нельзя угадать, нужно получить через API или посмотреть в Bitrix24.

## Поля, которые стоит завести в сделке

Можно начать с обычных полей `title`, `comments`, `sourceId`, `sourceDescription`, `contactIds`, `categoryId`, `stageId`, а детали события передавать в пользовательские поля.

Рекомендуемые пользовательские поля сделки:

- `MM Bot Person ID` - ID человека в нашей базе.
- `MM Bot Registration ID` - ID регистрации в нашей базе.
- `Telegram ID`.
- `Telegram username`.
- `Мероприятие`.
- `ID мероприятия`.
- `Формат участия` - online/offline.
- `Статус регистрации` - pending/approved/rejected/visited/cancelled.
- `Дата мероприятия`.
- `Facecast URL`.
- `Источник` - Telegram bot.

После создания полей Bitrix24 вернет технические коды вроде:

```text
ufCrm2_...
```

или при `useOriginalUfNames=Y`:

```text
UF_CRM_2_...
```

Эти коды нужно сохранить в `.env`.

## Триггеры синхронизации в нашем боте

1. После завершения анкеты:
   - найти/создать контакт в Bitrix24;
   - создать или обновить сделку “Анкета заполнена”.

2. После выбора онлайн:
   - создать/обновить контакт;
   - создать/обновить сделку по регистрации;
   - стадия: “Онлайн зарегистрирован”;
   - формат: online;
   - мероприятие, дата, Facecast URL.

3. После выбора офлайн:
   - создать/обновить контакт;
   - создать/обновить сделку по регистрации;
   - стадия: “Офлайн на проверке”;
   - формат: offline.

4. После аппрува офлайн:
   - обновить сделку;
   - стадия: “Офлайн подтвержден”.

5. После отказа офлайн:
   - обновить сделку;
   - стадия: “Отклонен” или “Предложен онлайн”.

6. После отметки ресепшна:
   - обновить сделку;
   - стадия: “Пришел”.

## Защита от дублей

Для людей:

- искать контакт по email и телефону через `crm.duplicate.findbycomm`;
- если контакт найден, обновлять его;
- если не найден, создавать новый контакт.

Для сделок:

- хранить `bitrix_contact_id` у человека в нашей базе;
- хранить `bitrix_deal_id` у регистрации в нашей базе;
- дополнительно передавать `originatorId = megapolis_event_bot` и `originId = registration:{id}`.

Так повторные синхронизации не будут плодить новые сделки.

## Минимальный первый релиз

1. Добавить таблицы/поля для хранения Bitrix24 ID.
2. Добавить `Bitrix24Client`.
3. Добавить `Bitrix24SyncService`.
4. Синхронизировать:
   - анкету;
   - онлайн-регистрацию;
   - офлайн-заявку;
   - офлайн-аппрув;
   - отметку прихода.
5. В админке показывать статус синхронизации и последнюю ошибку.

## Что нужно от владельца Bitrix24

- URL портала Bitrix24.
- Входящий webhook с правом `crm`.
- Название созданной воронки.
- Список стадий воронки или разрешение получить их через API.
- Решение: создавать “Анкета заполнена” как отдельную сделку или хранить это только в контакте.
- Коды пользовательских полей, если поля уже созданы руками.
